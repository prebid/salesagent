# adcp SDK 4.3.0 → 5.7.0 — Model Changeset Reference

Reference notes for the SDK upgrade tracked in **[prebid/salesagent#1388](https://github.com/prebid/salesagent/issues/1388)**.
This records the model-level differences between `adcp==4.3.0` (AdCP spec 3.0.1) and
`adcp==5.7.0` (AdCP spec 3.1.0-beta.3) so the reconciliation work in #1388 is traceable
to concrete shape changes. The actionable task list lives in the issue; this is the
underlying diff.

## Why a minor spec bump touches so many Python types

The SDK is generated by `datamodel-code-generator`, which inlines each JSON-Schema
`$ref` into a per-message local class rather than emitting one shared type. In
`adcp.types.generated_poc`, 431 class names are defined in two or more modules
(`Account`×37, `Error`×35, `Brand`×33), and class names carry an ordinal suffix tied to
their position in the inlined expansion. Consequences for an upgrade:

- A wire-level-additive spec change can still **rename or renumber** the Python class we
  import (e.g. `SignalId18` → `SignalId5` with an identical field set).
- The same concept maps to many non-`isinstance`-matching classes, so we bind imports to
  specific deep paths and must re-verify identity on each bump.

This is why the upgrade is mechanical-but-careful rather than a one-line pin change, and
why #1388 mandates AST-based import enumeration and field-set verification.

## Type changes (same field, different type)

| `Model.field` | 4.3 | 5.7 | Impact |
|---|---|---|---|
| `Creative.action` | `CreativeAction` (enum) | `CreativeAction \| str` | equality vs the enum can return `False` for the bare string — **silent** |
| `GetProductsResponse.products` | `list[Product]` | `list[Product] \| None` | `len()` on `None` |
| `GetSignalsResponse.signals` | `Sequence[Signal]` | `Sequence[Signal] \| None` | iteration on `None` |
| `CreativeAsset` | flat model | `RootModel[CreativeAsset1 \| CreativeAsset2]` | dict access pattern changes |
| `create_mcp_webhook_payload()` return | `dict[str, Any]` | `McpWebhookPayload` | `**` unpacking |
| `create_mcp_webhook_payload()` 3rd arg | `result` | `task_type` | positional-arg meaning |
| response aliases (`CreateMediaBuySuccessResponse`, …) | direct class | `getattr()` dynamic resolve | mypy "invalid base class" (runtime OK) |

**Watch `Creative.action`.** When 5.7 returns the bare string `"created"`,
`c.action == CreativeAction.created` is `False` with no error. Code that counts
creatives by comparing against the enum will undercount silently — normalize to the
enum (or compare by `.value`) when reconciling.

## Structural changes

- **Sync response envelopes** collapsed to a minimal protocol shape; runtime compat
  relies on `extra='allow'` on the models.
- **`CreativeAsset`** is now a discriminated union — asset dict values are
  `list[AssetVariant]` tagged by `asset_type`, not flat dicts.
