## 1. User impact inventory

The L1a deploy is a **one-time forced re-login event** for every active admin session. Impacts:

### 1.1 Forced re-login (admin + tenant admins)
- **Who:** every user with an active `session=...` cookie at cutover (internal admins + tenant admins).
- **What they see:** a redirect to Google OAuth (or OIDC for per-tenant SSO) on their next request after deploy.
- **UX:** ~1-2 second OAuth bounce if user is already signed into Google; up to 20 seconds if they must re-enter credentials.
- **Who is NOT affected:** AdCP advertisers (authenticate via bearer `x-adcp-auth` tokens, not sessions).

### 1.2 In-progress form data loss
- **Who:** users in the middle of editing any admin form at cutover.
- **What they lose:** unsaved form state.
- **Mitigation:** announcement message tells users to save before the window.

### 1.3 Mid-signup tenant flow reset
- **Who:** tenants in the middle of the signup flow (signup_flow / signup_step session keys).
- **What they lose:** signup progress; must restart from step 1.
- **Impact:** rare (signup is infrequent); self-healing on retry.

### 1.4 OIDC test-flow sessions lost
- **Who:** tenant admins testing their OIDC configuration at cutover.
- **What they lose:** `oidc_test_tenant_id`, `oidc_test_flow` session state; must re-initiate OIDC test.
- **Impact:** minor; test flow is short.

### 1.5 OAuth bounce in-flight
- **Who:** users mid-OAuth-authorization (clicked "Sign in", at Google consent screen at cutover moment).
- **What happens:** Authlib reports `invalid_state` on callback; user must retry login.
- **Impact:** rare and self-healing.

### 1.6 Cron / scheduler restart
- **Who:** nobody user-facing; internal ops awareness.
- **What happens:** background schedulers restart with lifespan; in-flight tick work that hasn't persisted is lost (schedulers are tick-idempotent; next interval picks up).
- **Impact:** none user-visible; flagged for ops.

## 2. Audit log impact

Expect a flood of `auth.login.success` events in the 5-minute window after deploy (approximately N events where N = count of active admin sessions). **Mute the "auth success rate 100x baseline" alert** during the cutover window to prevent spurious paging.

## 4. Runbook — ops steps at deploy

1. **T-48h:** post announcement to Slack + send customer email.
2. **T-5min:** mute `auth.login.success` rate alert.
3. **T-0:** merge L1a PR + deploy.
4. **T+10min:** verify sessions are writing `adcp_session` cookie (check browser devtools or a test login).
5. **T+60min:** unmute `auth.login.success` alert (baseline will settle within an hour).
6. **T+24h:** scan audit logs for any user who has NOT re-authenticated (may indicate a stuck browser cookie); follow up if >5 users are missing.

## 5. Fallback procedure

If >10% of users report login failures within 15 minutes of deploy:
1. Check `request.session.get("user")` is being populated after OAuth callback.
2. Check `Set-Cookie: adcp_session=...` is being emitted on the callback response.
3. If systematic breakage: redeploy the pre-L1a image tag `v2.0.0-rc.L0` via `flyctl deploy --image registry.fly.io/salesagent:v2.0.0-rc.L0 --strategy rolling`. (Note: pre-L1a image was the last Flask-era admin; session cookie is `session=...`, not `adcp_session=...`. Users will NOT need to re-auth after rollback since the Flask cookie is still valid.)
