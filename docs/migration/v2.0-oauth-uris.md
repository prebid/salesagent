# v2.0 OAuth Redirect URIs — Runbook

**Landed at L0-26 `904eb868`. Authoritative source for Google Cloud Console registration.**

Status: active (v2.0.0 migration)
Owners: platform engineering
Canonical spec: [`.claude/notes/flask-to-fastapi/CLAUDE.md`](../../.claude/notes/flask-to-fastapi/CLAUDE.md) §Critical Invariants #6

## TL;DR

The v2.0 Flask→FastAPI migration preserves **every OAuth redirect URI
byte-for-byte**. Any URI change — case, trailing slash, path segment,
or prefix — produces `redirect_uri_mismatch` from the identity provider
and renders login broken across a rolling deploy.

No update to Google Cloud Console registration, per-tenant OIDC
provider configuration, or GAM OAuth registration is required for v2.0.

## The three byte-immutable URIs

| Purpose | URI | File / guard |
|---|---|---|
| Super-admin Google OAuth | `/admin/auth/google/callback` | `src/admin/blueprints/auth.py`, byte-guard `tests/unit/test_oauth_redirect_uris_immutable.py` |
| Tenant OIDC callback | `/admin/auth/oidc/callback` | `src/admin/blueprints/oidc.py:209,215` (**no `{tenant_id}` segment** — tenant context lives in the session, not the URL) |
| GAM OAuth | `/admin/auth/gam/callback` | `src/admin/blueprints/auth.py:931,959` (**includes `/admin` prefix** — part of the registered URI) |

The byte-immutability guard (`tests/unit/test_oauth_redirect_uris_immutable.py`) asserts these exact strings appear in source; adding, renaming, or path-rewriting any of them trips the guard at `make quality` time.

## Why this matters

OAuth redirect URIs are registered with the identity provider at
account-setup time:

* **Google Cloud Console** (super-admin login + GAM) — a fixed allowlist
  per OAuth client, maintained manually in the console UI.
* **OIDC providers** (per-tenant) — configured by the tenant admin in
  their IdP's admin UI; often an OpenID Connect issuer with a preset
  `redirect_uris` array.

If the app emits a redirect URI that isn't on the allowlist, the IdP
returns `redirect_uri_mismatch` at the authorization-code exchange step.
There is no silent fallback — login is dead until either the app or the
IdP registration is changed.

## During the v2.0 rolling deploy

The migration keeps all three URIs at their current paths. Specifically:

* The `/admin/auth/google/callback` path stays on `/admin/...` even
  after the D1 (2026-04-16) canonical `/tenant/{tenant_id}/...` route
  decision — OAuth paths are in the D1 "stay on `/admin/*`" exception
  list because the pre-auth flow has no session-tenant to redirect with.
* `LegacyAdminRedirectMiddleware` (landing at L1c) explicitly
  allowlists `/admin/auth/*` so legacy `/admin/...` requests for OAuth
  endpoints are NOT 308-redirected to `/tenant/{tenant_id}/...`.
* The OIDC callback URI intentionally does NOT include `{tenant_id}` —
  tenant context is carried on the transit cookie set at the
  authorization step, resolved at callback time from session state.

## If an IdP registration needs to change (out of v2.0 scope)

This runbook covers the v2.0 migration surface only. If a future change
intentionally moves a redirect URI:

1. Register the NEW URI in the IdP **before** rolling out the code.
   Both URIs can be on the allowlist simultaneously during the cutover.
2. Roll out the code change.
3. Verify logins succeed on the new URI.
4. Remove the OLD URI from the IdP allowlist.
5. Update `tests/unit/test_oauth_redirect_uris_immutable.py` to the new
   expected strings.
6. Update this runbook.

Never step (1)→(2) out of order; never remove the old URI before (3).

## Verification

Before cutting any v2.0 layer PR that touches admin auth:

```bash
uv run pytest tests/unit/test_oauth_redirect_uris_immutable.py -v
```

This is part of `make quality`; failure blocks merge.
